# PR05-A 请求生命周期与卡死修复

## 问题

对话请求会依次执行资料决策、可选联网搜索、主回答和可选续写。此前网络层读取超时较长，上层没有业务级总超时或用户取消入口；同时角色级异常只记录在编排结果中，ViewModel 未向用户反馈，因此可能长时间显示“正在思考”，最终仍没有回复或错误信息。

## 实现

- ViewModel 保存当前圆桌运行 Job，拒绝重复触发。
- 输入区在运行时提供“停止生成”按钮。
- 单角色回答最长等待 120 秒，超时后清理 Pending，并继续后续角色。
- 整轮最长等待 8 分钟，超时后停止并保留已完成回复。
- 用户取消、角色超时和普通异常都会清理 Pending 消息。
- `CancellationException` 在网络重试层和编排层直接向上传播，不会被包装成网络失败或触发换 Key。
- 编排结果新增超时角色列表；ViewModel 对全部失败、部分失败和超时给出可操作反馈。
- App 启动时清理上次进程异常中断遗留的 Pending；新请求必须等待该清理完成，避免异步清理误删新 Pending。
- 没有可用 Key 或没有启用角色时立即给出提示。

## 自动测试

- 单角色超时后清理 Pending，并继续后续角色。
- 用户取消后清理 Pending，且取消异常向上传播。
- 网络重试引擎不捕获或包装取消异常。
- 全部超时、部分失败和正常完成的用户反馈。
- 现有圆桌顺序、预算、Key 调度和数据库迁移测试继续保留。

## 不包含

- 流式输出。
- 失败角色原位重试。
- 错误类型细分展示。
- Room 版本或 Schema 变更。
- API Key 存储、签名、包名或 Release 配置变更。
