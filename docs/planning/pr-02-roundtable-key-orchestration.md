# PR 02：圆桌调度与 API Key 编排重构

> 优先级：P0
> 覆盖：Key 分组、重试策略、Interaction 会话一致性、圆桌语义

## 目标
让实现与产品描述一致：角色如何发言、如何共享上下文、如何消耗 Key 都必须可预测。

## 任务清单

### 1. 明确圆桌模式
二选一：
- 串行圆桌：后一个角色看到前面发言。
- 并行专家面板：最后增加综合阶段。

禁止继续保持文档与代码不一致。

### 2. 重构 Key 调度
- 引入 Key Lease/Reservation 概念。
- 调度层决定 Key。
- 网络层禁止偷偷重新选择 Key。

### 3. 重构失败处理
错误分类：
- 400/404：请求错误，不换 Key。
- 401/403：Key 无效。
- 429：退避和冷却。
- 5xx：有限重试。

### 4. 修复并发竞态
- Interaction ID 按角色或链路隔离。
- 避免多个协程覆盖同一会话状态。

### 5. 增加预算控制
- 单轮最大调用数。
- 最大角色数量。
- Token 上限。

## 验收标准
- Key 分配日志与实际请求一致。
- 重试不会无限消耗 Key。
- 圆桌行为稳定可复现。
- 单元测试覆盖调度逻辑。

## 禁止事项
- 不通过随机延迟规避平台限制。
- 不隐藏实际调用数量。
