# ADR 005: 基于三模型级联与 Google Search Grounding 的实时联网决策架构

## 状态
已通过 (Accepted)

## 背景
随着圆桌会议机制向更宽阔的垂直领域演进，用户经常会提出涉及最新事实、时效性热点或前沿研究的问题。由于现存的知识库（`references/` 和 `examples/`）是离线静态打包的，纯依靠本地知识容易在时效性话题上产生幻觉。

此外，用户希望能够在高比例下引入最新的互联网知识，并希望自主控制联网行为。同时，需要支持针对单一问题拆解出多个子查询并依次搜索，以提供更强的接地广度。

## 决策
我们决定将原本的“双模型级联”升级为“三模型级联联网搜索配合”，具体方案为：

1. **三模型分工**：
   - **模型一 (3.1lite)**：`gemini-3.1-flash-lite-preview`。接收脑暴上下文与本地候选文件。输出 JSON，双决策决定需要加载的本地文件及生成的 1 到多个联网 `searchQueries`。
   - **模型二 (2.5flash)**：`gemini-2.5-flash`（免费层）。当决定联网时，针对这组 Query，配置 `google_search` 联网工具（REST `tools` 字段下传入 `google_search`）依次拉取网络总结，并解析返回中的 `groundingMetadata` 拿到真实的网页标题 and 链接。
   - **模型三 (3.5flash)**：`gemini-3.5-flash`。接收本地知识资料与格式化后的联网接地资料（包含引用链接），开启高思考度进行整合，生成最终圆桌作答。

2. **联网模式控制 (SearchMode)**：
   - **SMART (智能搜索)**：由 3.1lite 智能决策是否联网及搜索内容。
   - **FORCE (强制联网)**：强行执行联网。3.1lite 被引导生成至少 1 个 query，若为空则自动通过用户提问兜底，保证至少 1 次 2.5flash 联网调用。
   - **OFF (关闭联网)**：完全不调用联网，只使用本地知识。

3. **健壮的降级容错**：
   - 对 3.1lite 的 JSON 解析失败时，内置正则表达式进行降级提取，防止非标格式导致崩溃。
   - 联网过程由协程在后台按顺序或并发索取，对于各请求异常进行捕获并熔断，防止由于单次网络波动阻断整个圆桌流程。

## 影响
- **优点**：
  - 极大地提升了圆桌会议的实时客观事实准确度，能够当场获取 2026 年最新事件和参考数据，且自动带有链接引用。
  - 为用户提供了极佳的自适应（智能）与确定性（强制）的控制体验，能将额度与质量有效权衡。
  - 维持了 implicit context caching 的高命中，本地合并逻辑避免了本地向量库对 App 包体的压力。
- **缺点**：
  - 单次角色发言若触发联网会产生多次 API 级联调用，略微增加了首字 TTFT 延迟与 Key 池额度消耗，但可以通过内置多 Key 随机平摊和熔断机制进行完美消化。
